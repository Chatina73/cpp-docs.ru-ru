# <a name="overview-of-windows-programming-in-c"></a>Общие сведения о программировании на C++ в Windows

Существует несколько основные категории приложений Windows, которые можно создать с помощью C++. Каждый имеет собственную модель программирования и набор библиотек, относящиеся к Windows, но стандартной библиотеки C++, а также сторонних библиотек C++ можно использовать в любой из них. 

## <a name="command-line-console-applications"></a>Приложения командной строки (консоль)

C++ консольные приложения, запустите из командной строки в окне консоли и может отображать только выходного текста. Дополнительные сведения см. в разделе [консольных приложений](console-applications-in-visual-cpp.md).
 
## <a name="native-desktop-client-applications"></a>Собственный клиент для настольных приложений

Термин *приложения собственного клиента рабочего стола* ссылается на оконные приложение C или C++, который использует исходный машинный код [API-интерфейсы C Windows и/или API-интерфейсы COM](/windows/desktop/apiindex/windows-api-list) для доступа к операционной системе. Эти API написаны главным образом на языке C. При создании такого приложения, вы можете выбрать напрямую программировании в стиле C цикл обработки сообщений, обрабатывает события операционной системы, или с помощью *Microsoft Foundation Classes* (MFC), библиотеку C++, который заключает в оболочку Win32 в результате которого является довольно объектно ориентированным. Оба подхода считается «современные» по сравнению с универсальной платформы Windows (см. ниже), но и по-прежнему полностью поддерживается и содержат миллионы строк кода, запускающие сейчас в мире. Приложения Win32, запускаемые в окне требует, чтобы разработчик явно работать с сообщениями Windows внутри процедуры функции Windows. Несмотря на название приложения Win32 можно скомпилировать в виде 32 (x86) или 64-разрядных (x64) двоичный. В Visual Studio IDE условия x86 и Win32 являются синонимами.

Чтобы начать работу с традиционной программирования Windows на C++, см. в разделе [начало работы с Win32 и C++](/windows/desktop/LearnWin32/learn-to-program-for-windows). После получения понимания того, Win32, удобнее получить сведения о [настольные приложения MFC](/mfc/mfc-desktop-applications). Пример традиционного C++ настольным приложением, которое использует сложные графики, см. в разделе [Hilo: Разработка приложений C++ для Windows](https://msdn.microsoft.com/library/windows/desktop/ff708696.aspx).

### <a name="c-or-net"></a>C++ или .NET? 

Для сценариев наиболее настольных приложений (другими словами, не предназначенных для универсальной платформы Windows), рассмотрите возможность использования C# для создания пользовательского интерфейса. Это обусловлено программирования .NET обычно менее сложный и менее подверженным ошибкам и имеет более современный объектно ориентированный API Win32 или MFC. В большинстве случаев его производительность более чем достаточно. Функции .NET Windows Presentation Foundation (WPF) для вывода разнообразной графики, и вы можете использовать Win32, а также современных Windows API среды выполнения (см. ниже универсальной платформы Windows). Как правило рекомендуется использовать C++ для классических приложений, когда требуется:

- точно контролировать использование памяти
- важнейшими экономии энергопотребления
- Использование графического процессора для общие вычисления
- доступ к DirectX
- При интенсивном стандартных библиотек C++

Можно создать пользовательский интерфейс в C# и использовать C++выполняет для включения в собственном приложении C++ библиотеки. Дополнительные сведения см. в разделе [программирование .NET с использованием C++выполняет](../dotnet/dotnet-programming-with-cpp-cli-visual-cpp.md).

## <a name="com-components"></a>COM-компонентов

[Объектов модели компонентов (COM)](/windows/desktop/com/the-component-object-model) — это спецификация, включает программы, написанные на разных языках для взаимодействия друг с другом. Многие Windows, компонентов, реализуются в виде COM-объекты и следовать стандартным правилам COM, для создания объекта интерфейса уничтожения обнаружения и объекта.  С помощью COM-объекты от классических приложений C++ очень простой, но более сложная написания собственных COM-объекта. [Active Template Library (ATL)](../atl/atl-com-desktop-components.md) предоставляет макросы и вспомогательные функции, которые упрощают разработку COM. Дополнительные сведения см. в разделе [классических компонентов ATL COM](../atl/atl-com-desktop-components.md).

## <a name="windows-universal-apps"></a>Универсальные приложения Windows

Универсальной платформы Windows (UWP) — это современный API Windows. Приложения UWP запуск на любом устройстве Windows 10, использовать XAML для пользовательского интерфейса и полностью поддержкой сенсорного ввода. Дополнительные сведения о UWP см. в разделе [что такое приложение универсальной платформы Windows (UWP)?](/windows/uwp/get-started/whats-a-uwp) и [руководстве по универсальным приложениям Windows](/windows/uwp/get-started/universal-application-platform-guide).

Исходный C++ поддержка универсальной платформы Windows состоял из (1) C++/CX, диалекте C++ с помощью синтаксиса расширения или (2 Windows библиотеки среды выполнения (WRL) который основан на стандарте C++ и COM. Оба C++/CX и WRL по-прежнему поддерживаются. Для новых проектов рекомендуется [ C++/WinRT](/windows/uwp/cpp-and-winrt-apis/intro-to-using-cpp-with-winrt) полностью основанная на стандарте C++ и обеспечивает более высокую производительность. 

## <a name="desktop-bridge"></a>Мост для классических приложений

В Windows 10 можно упаковать вашей существующего классического приложения или COM-объект как приложение универсальной платформы Windows и добавить компоненты универсальной платформы Windows, например сенсорный или вызов API из современных набор Windows API. Также можно добавить приложения универсальной платформы Windows для настольных решений в Visual Studio и пакета их вместе в один пакет и использовать API-интерфейсы Windows для связи между ними.

В Visual Studio 2017 версии 15.4 или более поздней версии можно создать проект пакета приложения Windows для значительно упрощают работу по упаковки существующего классического приложения. Накладывается несколько ограничений по отношению к вызывает какие реестра или использует API-интерфейсы приложения для настольных компьютеров, но во многих случаях можно создать альтернативный путей для реализации подобной функциональности во время выполнения в пакет приложения. Дополнительные сведения см. в статье [Мост для классических приложений](/windows/uwp/porting/desktop-to-uwp-root).

## <a name="games"></a>Игры

Игры DirectX можно запустить на ПК или Xbox. Дополнительные сведения см. в разделе [DirectX Graphics and Gaming](/windows/desktop/directx).

## <a name="sql-server-database-clients"></a>Клиенты базы данных SQL Server

Чтобы получить доступ к базам данных SQL Server из машинного кода, используйте ODBC или OLE DB. Дополнительные сведения см. в разделе [SQL Server Native Client](/sql/relational-databases/native-client/odbc/sql-server-native-client-odbc).

## <a name="windows-device-drivers"></a>Драйверы устройств Windows

Драйверы, низкоуровневые компоненты, обеспечивающие доступность для приложений данных из устройств и других компонентов операционной системы. Дополнительные сведения см. в разделе [набор драйверов Windows (WDK)](/windows-hardware/drivers/index).

## <a name="windows-services"></a>Службы Windows

Windows *службы* — это программа, которое может выполняться в фоновом режиме с незначительной или нулевой взаимодействия с пользователем. В UNIX это называется *управляющие программы*. Дополнительные сведения см. в разделе [служб](/windows/desktop/services/services).

## <a name="sdks-libraries-and-header-files"></a>Пакеты SDK, библиотеки и заголовочные файлы

Visual Studio включает библиотеки времени выполнения C (CRT), стандартной библиотеки C++ и другие библиотеки Microsoft. Папки включения, содержащие файлы заголовков для этих библиотек расположены в каталоге установки Visual Studio в папке \VC\, или в случае CRT, в папке установки пакета Windows SDK.

Можно использовать [диспетчером пакетов Vcpkg](../build/vcpkg.md) удобно установить сотни сторонних библиотек с открытым кодом для Windows.

Библиотеки Microsoft:

- Microsoft Foundation Classes (MFC): Объектно ориентированная платформа для создания традиционных программ Windows — особенно корпоративные приложения —, со сложным пользовательским интерфейсом, включающим кнопки, списки, представления в виде дерева и других элементов управления. Дополнительные сведения см. в разделе [MFC Desktop Applications](../mfc/mfc-desktop-applications.md).

- Библиотека шаблонных (классов ATL): Многофункциональная вспомогательная библиотека для создания COM-компонентов. Для получения дополнительной информации см. [ATL COM Desktop Components](../atl/atl-com-desktop-components.md).

- C++ AMP (C++ Accelerated Massive Parallelism): Библиотека, позволяющая высокопроизводительные вычислительные задачи общего характера в графическом Процессоре. Для получения дополнительной информации см. [C++ AMP (C++ Accelerated Massive Parallelism)](../parallel/amp/cpp-amp-cpp-accelerated-massive-parallelism.md).

- Среда выполнения с параллелизмом: Библиотека, упрощающая параллельное и асинхронное программирование для мультиядерных и многоядерных устройств. Для получения дополнительной информации см. [Concurrency Runtime](../parallel/concrt/concurrency-runtime.md).

Для многих сценариев программирования Windows также требуется пакет Windows SDK, в который входят файлы заголовков, обеспечивающие доступ к компонентам операционной системы Windows. По умолчанию Visual Studio устанавливает пакет SDK Windows, как компонент рабочей нагрузки C++ Desktop, который обеспечивает возможность разработки приложений универсальной Windows. Для разработки приложений универсальной платформы Windows, требуется Windows 10 версии пакета SDK для Windows. Сведения см. в разделе [пакета SDK для Windows 10](https://dev.windows.com/downloads/windows-10-sdk). (Дополнительные сведения о пакетах Windows SDK для более ранних версий Windows см. в разделе [архиве пакета SDK для Windows](https://developer.microsoft.com/windows/downloads/sdk-archive)).

**Программа Files (x86) \Windows комплекты** является местоположением по умолчанию для всех версий пакета SDK Windows, который вы установили.

Другие платформы, например Xbox и Azure, обладают собственными пакетами SDK, которые вам, возможно, потребуется установить. Дополнительные сведения см. в Центре разработчика DirectX и в Центре разработчика Azure.
